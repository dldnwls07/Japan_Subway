# 개발 문서 인덱스

일본 지하철역 이름 타자 게임 애플리케이션 — 구현 상세 명세 모음.

**상위 문서**: `일본_지하철역_타자게임_개발계획서_개정판.docx` (전체 아키텍처, 일정, 리스크, 벤치마킹, 예산 등 계획 수준 내용)

이 `docs/` 폴더는 계획서 검토 과정에서 "실제 구현을 시작하기엔 구체성이 부족하다"고 판단된 5개 영역을 코드 수준까지 내려서 작성한 것이다. 계획서가 **왜/무엇을** 다룬다면, 이 폴더는 **어떻게**를 다룬다.

## 읽는 순서 (착수 전 최초 1회)

1. 계획서 1장(범위·일정) — Phase 0가 무엇인지 확인
2. `translation-engine-spec.md` — Phase 0의 핵심(번역 엔진)부터 스파이크
3. `mobile-spec.md`의 웹 섹션 — 게임 루프의 입력 처리 기반 마련
4. `backend-spec.md` — 랭킹이 필요해지는 시점(Phase 1)에 착수
5. `devops-deployment.md` — 로컬 환경 세팅 시 참고, 배포 직전 재확인
6. `api-spec.md` — 프론트-백엔드 계약이 바뀔 때마다 갱신

## 이 문서들의 성격

- 여기 담긴 코드는 **실제 동작이 검증된 코드가 아니라 구현 골격(skeleton)**이다. 특히 `translation-engine-spec.md`의 매핑 테이블 값과 `mobile-spec.md`의 디바운스 타이밍은 실기 테스트로 조정이 필요하다고 각 문서에 명시해 두었다.
- 각 문서 끝에 "미결 사항" 또는 그에 준하는 항목이 있으면, 코딩을 시작하기 전에 먼저 결정하고 이 문서를 갱신한다.
- 코드와 문서가 어긋나기 시작하면 문서가 아니라 코드를 신뢰하되, PR 리뷰에서 문서 갱신 여부를 체크리스트에 넣을 것을 권장한다(2.2절 PR 프로세스에 추가 검토 항목으로 반영 가능).

## 아직 다루지 않은 것

- 관리자용 검수 큐(`is_flagged`) 화면 — backend-spec.md §10 미결 사항 참고
- 소셜 로그인 확장 시의 계정 이관(닉네임 기반 → OAuth 병합) 정책
- 개인정보처리방침 실제 문구(9.4절 요구사항은 정의됐으나 법적 문서 자체는 별도 작성 필요)
- 다국어 UI 문자열(i18n) 관리 방식 — 현재 계획서는 번역 데이터 파이프라인만 다루고 UI 자체의 다국어화는 미정
